# 2026-06-01 Stage 0.9 API（管理画面向け）

## 背景
`008_api_admin.md`。施策の run/approve/send/results を公開する管理 API と、横断的関心事（Idempotency-Key・テナントスコープ・認証スタブ・OpenAPI）を実装する。Controller はビジネスロジックを持たず Application Service を呼ぶだけ（CLAUDE.md §4.2）。

## 実施内容
- **Application（読み取り追加）**: `IMatchingCampaignQueries` / `MatchingCampaignQueries`、`CampaignResultsView` / `CandidateScoreView`（PII・顧客識別子を含まない結果概況）。
- **Api/Controllers**: `MatchingCampaignsController`（`POST /api/campaigns/run` / `POST /api/campaigns/{id}/approve` / `POST /api/campaigns/{id}/send` / `GET /api/campaigns/{id}/results`）。`Result`→HTTP 写像（not_found→404, not_approved/invalid_state→409, その他検証→422）。
- **Api/Idempotency**: `IIdempotencyStore` + `InMemoryIdempotencyStore`（24h TTL）、`IdempotencyRecord`、`IdempotencyFilter`（`IAsyncActionFilter`）。キー欠如→400、同一キー＋同一本文→保存済みレスポンス再生（再実行しない）、同一キー＋異なる本文→409。キーはテナントでスコープ。5xx・例外はキャッシュしない。
- **Api/Tenancy**: `RequestContext`（`ITenantContext` 実装・スコープ付き・操作者も保持）、`TenantResolutionMiddleware`（認証スタブ：`/api` 配下で `X-Tenant-Id` を要求、未指定/不正→401、`X-User-Id` を任意解決）。
- **Api/Time**: `SystemClock`（`IClock`）。
- **Api/Contracts**: `ApiError` / `RunCampaignRequest` / `RunCampaignResponse` / `ApproveCampaignRequest` / `CampaignResultsResponse` / `CandidateScoreResponse`。
- **Program.cs**: Controllers・OpenAPI・Scalar UI・DI（IClock/RequestContext/ITenantContext/IIdempotencyStore/IdempotencyFilter）・テナント解決ミドルウェアを配線。
- **依存追加**: `Scalar.AspNetCore` 2.14.14（Api）、`Microsoft.AspNetCore.Mvc.Testing` 10.0.8（Api.Tests）。
- **テスト（WebApplicationFactory）**: `MatchingCampaignsApiTests` 5 件＋テストダブル（`CampaignApiFactory` でデータ系ポートをインメモリ差し込み、テナントスコープ付きリポジトリ等）。

## 設計判断
- **OpenAPI / Swagger UI（DoD との差異・ユーザー確認済み）**: 既存 `Program.cs` は「Swagger UI は Phase 1 で追補」と Microsoft.AspNetCore.OpenApi を採用。DoD「/swagger に日本語説明」と食い違うため確認し、**Scalar UI を採用**（`/scalar`）。XML コメント由来の日本語説明が `/openapi/v1.json` に出力されることを実機確認（run の summary に「施策を実行（候補抽出＋スコアリング）…」）。Swashbuckle は導入しない。
- **施策サービスの DI 登録を本 Stage では繰り延べ（重要）**: `IMatchingCampaignService` / `IMatchingCampaignQueries` は依存データ系ポート（リポジトリ/候補ソース/ポリシー/AI/Outbox/UnitOfWork）の実装が未存在。Program.cs で登録すると Development の ServiceProvider 検証（ValidateOnBuild）で起動失敗するため、**登録は実装が揃う後続 Stage（Infrastructure / 0.10 / 0.11）に繰り延べ**、本 Stage の検証は WebApplicationFactory のテスト host にインメモリ実装を差し込んで実施。アプリ自体は起動可能（/health・/scalar・/openapi 稼働）。それまで `/api/campaigns` は 500 を返す旨を Program.cs と本記録に明記。
- **認証はスタブ**: `X-Tenant-Id` / `X-User-Id` ヘッダ。Phase 1 の本認証（OIDC/JWT・ロール）に置換前提。テナント未解決時は安全側で 401。
- **冪等ハッシュ**: バインド済み Action 引数を JSON 化して SHA-256。`CancellationToken` 等の非シリアライズ対象は除外。
- **結果レスポンスの PII 非含**: `GET results` は件数・状態・候補スコア（Offer/枠）のみ。連絡先・顧客識別子を返さない（CLAUDE.md §9.3, §11, DoD）。
- **承認者の解決**: `ApproveCampaignRequest.ApprovedBy`（省略可）→ 省略時は認証コンテキストの操作者（`X-User-Id`）。両方なければ Application が `approver_required`（422）。
