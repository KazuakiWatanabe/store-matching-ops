# ADR-0010: 値引き上限と配信制御

- ステータス: 採用
- 日付: 2026-05-31
- 関連: CLAUDE.md §4.3, 設計 §6.3/§9, タスク `004_offer_and_catalog`

## コンテキスト
クーポン等の値引き提案は、店舗設定の上限を超えてはならない（過剰値引きの防止）。
また配信は頻度上限・オプトアウトを尊重する必要がある。これらを「守られていることを保証できる」形で
Domain の不変条件として表現したい。

## 決定
### 値引き上限 (DiscountCap)
- 値引きと上限は **金額 (Money) または率 (0〜1 の decimal)** のいずれかで表す（`DiscountKind`）。
- `DiscountCap.EnsureWithin(Discount)` が、提案された値引きが上限を超えないことを機械的に保証する。
  - 種別不一致（金額上限に率提案 等）→ `DomainException`。
  - 金額の場合は通貨一致を要求し、`Amount` 比較で超過を拒否。
  - 率の場合は `Rate` 比較で超過を拒否。
- **クーポンは値引き上限を必須で保持**する（`Offer.CreateCoupon`）。メニュー/コースは上限を持たず（`Offer.CreateItem`）、値引き検証を呼ぶと `DomainException`。
- v0 では同種比較に限定する（金額上限 vs 率提案の換算は行わない）。換算が必要になれば本 ADR を改訂する。

### 有効状態・適用条件
- `Offer.IsActive` と `OfferConditions`（適用曜日・対象セグメントタグ）を保持。
- `Offer.IsAvailableOn(date)` は「有効かつ曜日条件を満たす」を返し、無効・条件外の Offer を提案候補から除外する基礎とする（結合検証は 005）。
- 対象セグメントは Customers のセグメント方針（タグベース）に整合させる。

### 配信制御（頻度上限・オプトアウト）の所在
- オプトアウトは `Customer`（Stage 0.3）で、通知頻度上限は Notifications（Stage 0.9）で強制する。
- 本 ADR は **値引き上限の Domain 強制**を定める。頻度・オプトアウトと併せて「配信制御」を構成する。

## 影響
- 値引き上限超過の提案は Domain で作れない（テスト必須観点）。
- 金額上限は通貨整合を前提とする（v0 は JPY 想定）。
- 率と金額の混在換算は将来課題（ADR 改訂で対応）。
